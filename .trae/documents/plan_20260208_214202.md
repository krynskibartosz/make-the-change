# Plan d'Activation des Données Dormantes

Ce plan se concentre exclusivement sur l'utilisation des tables existantes (`investment`, `identity`, `content`) pour dynamiser le frontend, sans créer de nouvelles structures en base.

## 1. Actualités des Projets (Priorité Haute)
*   **Source :** Table existante `investment.project_updates` (actuellement inexploitée).
*   **Objectif :** Afficher la vie du projet (photos, avancement) sur la page détail.
*   **Tâches :**
    1.  Créer une Server Action ou une requête dans `projects/[slug]/page.tsx` pour récupérer les updates liés au projet.
    2.  Créer un composant UI `ProjectTimeline` ou `ProjectNews` pour afficher ces actualités (date, titre, contenu, images).
    3.  Intégrer ce composant dans la page de détail du projet.

## 2. Conformité & Préférences (Priorité Moyenne)
*   **Source :** Table existante `identity.user_consents` (actuellement vide).
*   **Objectif :** Rendre l'application conforme en traçant les consentements réels.
*   **Tâches :**
    1.  Modifier l'action d'inscription (`register/actions.ts`) pour insérer une entrée dans `user_consents` lors de la création du compte.
    2.  Ajouter un switch dans la page `settings` du dashboard pour permettre à l'utilisateur de révoquer/donner son consentement (ex: marketing).

## 3. Blog & Contenu (Nettoyage)
*   **Source :** Table existante `content.blog_posts`.
*   **Objectif :** Garantir que tout le contenu est dynamique.
*   **Tâches :**
    1.  Vérifier `blog/page.tsx` et supprimer tout fallback sur `mock-posts.ts`.
    2.  S'assurer que la récupération des images utilise bien le champ `blurhash` de la table `media_assets` pour la performance.

## 4. Historique Financier (Bonus)
*   **Source :** Schema `finance` (Grand Livre).
*   **Objectif :** (Optionnel) Afficher un historique plus précis si le besoin se fait sentir, en remplaçant la vue simplifiée par une requête sur `journal_entries`.
